# 워크플로우 규칙

## 이슈 규칙

- 이슈는 반드시 `.github/ISSUE_TEMPLATE` 아래 템플릿을 기준으로 작성합니다.
- 작업 성격에 맞는 템플릿을 선택해 작성하고, 템플릿 구조를 임의로 무시하지 않습니다.
- 이슈 제목은 반드시 `[Feat] 기능명` 형식을 사용합니다.
- 이슈 제목에 사용할 수 있는 키워드는 `Feat`, `Refactor`, `Design`, `Fix`, `Docs`, `Test`, `Chore`만 허용합니다.
- 이슈 제목 규칙은 반드시 지켜야 합니다.
- 이슈 템플릿의 `TO-DO` 체크리스트는 기본 예시라 처음부터 체크된 상태로 제공됩니다.
- 새 이슈를 생성할 때도 템플릿의 `TO-DO` 체크리스트는 기본적으로 체크된 상태를 유지한 채 작성합니다.
- 따라서 체크 여부만 보고 작업 완료로 해석하지 말고, 실제 작업에 맞게 항목과 체크 상태를 직접 수정해야 합니다.

## 브랜치 규칙

- 모든 작업 브랜치는 반드시 `develop` 브랜치에서 파생되어야 합니다.
- 브랜치 이름은 반드시 `키워드/기능명-소문자-케밥케이스/이슈번호` 형식을 사용합니다.
- 브랜치 예시: `feat/agent-skills/7`
- 저장소 내 새 폴더명도 특별한 이유가 없다면 케밥케이스를 사용합니다.
- 브랜치에 사용할 수 있는 키워드는 `feat`, `refactor`, `design`, `fix`, `docs`, `test`, `chore`만 허용합니다.
- 위 브랜치 네이밍 규칙은 반드시 지켜야 합니다.

## 커밋 규칙

- 커밋 메시지는 반드시 `키워드: (#이슈번호) 작업내용` 형식을 사용합니다.
- 커밋 예시: `feat: (#5) 로그인 페이지 UI 작성`
- 커밋에 사용할 수 있는 키워드는 `feat`, `refactor`, `design`, `fix`, `docs`, `test`, `chore`만 허용합니다.
- 커밋 메시지 규칙은 반드시 지켜야 합니다.

## PR 규칙

- PR의 base 브랜치는 반드시 `develop`으로 설정합니다.
- `main` 브랜치는 전체 개발 완료 후에만 병합 대상으로 사용합니다.
- PR 본문은 반드시 `.github/pull_request_template.md`를 기준으로 작성합니다.
- PR 템플릿의 구조와 항목을 임의로 생략하지 않습니다.
- PR 제목은 반드시 `[Feat] 기능명` 형식을 사용합니다.
- PR 제목의 기능명은 이슈 제목과 반드시 동일해야 합니다.
- PR 제목 규칙도 반드시 지켜야 합니다.
- 기본 머지 전략은 `Rebase and merge`를 사용합니다.
- 브랜치 안 커밋 수를 줄여 한 개의 커밋으로 정리할 필요가 분명한 경우에만 예외적으로 `Squash and merge`를 사용합니다.
- `Merge commit` 방식은 사용하지 않습니다.

## PR 메타데이터 규칙

- PR 생성 후에는 담당자를 반드시 지정합니다.
- 담당자 지정은 GitHub UI 또는 `gh pr edit --add-assignee <github-id>` 명령으로 처리할 수 있습니다.
- PR 생성 후에는 이슈 템플릿 유형과 동일한 라벨을 반드시 지정합니다.
- PR 라벨은 `.github/ISSUE_TEMPLATE`에 정의된 실제 라벨명을 그대로 사용합니다.
- 사용 가능한 PR 라벨은 `✨ Feature`, `🛠️ Refactor`, `🎨 Design`, `🐞 Fix`, `📘 Docs`, `✅ Test`, `🔧 Chore`입니다.
- 라벨 지정은 GitHub UI 또는 `gh pr edit --add-label "<label-name>"` 명령으로 처리할 수 있습니다.
- PR 생성 후에는 project와 milestone도 반드시 지정합니다.
- project는 이 저장소에 연결된 유일한 프로젝트를 사용합니다.
- project의 `Status` 값은 항상 `Doing`으로 설정합니다.
- milestone은 open 상태인 마일스톤 중 가장 최근에 생성된 항목을 사용합니다.
