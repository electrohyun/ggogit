# 익명/게스트 공개 접근 및 렌더링 전략

## 목적

- 처음 방문한 사용자가 로그인 없이도 꼬깃의 학습, 챌린지, 커뮤니티 읽기 경험을 바로 체험할 수 있게 한다.
- 기록 저장, 콩, 스트릭, 배지, 커뮤니티 작성처럼 서버에 사용자 상태를 남기는 행동은 게스트 또는 로그인 사용자에게만 허용한다.
- 공개 읽기 페이지는 검색 엔진이 접근할 수 있게 하고, 추후 SSG/ISR 적용이 가능한 구조로 분리한다.

## 사용자 상태 용어

- 익명: 세션 없음. 읽기와 플레이 체험은 가능하지만 서버 기록 저장은 불가능하다.
- 게스트: Supabase anonymous auth 세션 있음. 랜덤 이름을 가지며 기록, 콩, 스트릭 등을 임시 저장할 수 있다.
- 로그인 사용자: 소셜 로그인 계정이 있으며 기록을 계정에 저장한다.

## 현재 합의된 방향

- 익명도 학습, 챌린지, 커뮤니티 읽기, 크레딧 접근이 가능해야 한다.
- 익명 헤더의 콩과 스트릭은 `0`으로 표시한다.
- 익명에게도 음소거 버튼과 사운드 설정을 제공한다.
- 익명 헤더의 프로필 자리에는 `시작하기` 버튼을 둔다.
- 게스트 세션은 자동 생성하지 않고, 사용자가 버튼을 누를 때 생성한다.
- 첫 방문 안내 모달은 띄우지 않는다.
- 저장, 보상 수령, 커뮤니티 작성 같은 행동 시에도 로그인/게스트 유도 모달을 띄운다.
- 익명 플레이 결과는 화면에 보여준다.
- 익명으로 달성한 현재 플레이 결과는 게스트 전환 또는 소셜 로그인 후 반영하는 방향으로 간다.
- 데일리 퀘스트는 익명에게도 보여주되, 수령하기를 누르면 로그인/게스트 유도 모달을 띄운다.
- 게스트 세션은 쿠키 기반 임시 세션이 아니라 Supabase anonymous auth로 만든다.

- 질문 상세 보기
- 크레딧 보기
- 사운드 온오프 및 볼륨 설정

### 게스트 또는 로그인 사용자 필요

## 로그인/게스트 유도 모달

기존 로그인 모달을 재활용하되, 상황별 문구와 꼬꼬 이미지를 바꾼다.

- 기록 저장: 스테이지 또는 챌린지 결과 저장을 위해 게스트 시작 또는 로그인을 권유한다.
- 보상 수령: 콩과 데일리 퀘스트 보상을 받으려면 기록을 남길 계정이 필요하다고 안내한다.
- 커뮤니티 작성: 글이나 댓글을 남기려면 작성자 정보가 필요하다고 안내한다.

## 익명 결과 반영

- 익명으로 푼 결과는 우선 `sessionStorage`에 임시 보관하는 방향을 검토한다.
- 게스트 시작 또는 소셜 로그인 완료 후 pending result를 서버에 반영한다.
- v0.5에서는 오래된 localStorage 기록 이월보다, 현재 세션에서 방금 달성한 결과 반영을 우선한다.

### 현재 구현 제약

- 오늘의 챌린지는 최종 결과 단위 저장 API가 있으므로 익명 결과를 게스트/로그인 전환 후 반영하는 구조로 확장할 수 있다.
- 미니 퀴즈 스테이지는 현재 `start_mini_quiz_stage_attempt`, `submit_mini_quiz_question_answer` RPC가 로그인 사용자 기준으로 문제별 attempt를 생성/채점한다.
- 따라서 익명으로 이미 끝낸 미니 퀴즈 결과를 사후에 그대로 저장하려면 별도 trusted server action/RPC가 필요하다.
- 이번 1차 구현은 익명에게 플레이와 결과 확인을 열고, 저장 버튼을 눌렀을 때 로그인/게스트 유도 모달을 띄우는 데서 멈춘다.
- 미니 퀴즈 결과 사후 저장은 DB/RPC 설계가 필요하므로 같은 이슈의 후속 커밋 또는 별도 범위에서 처리한다.

### 미니 퀴즈 사후 저장 RPC 초안

- SQL 파일: `supabase/sql/019_claim_mini_quiz_stage_result_from_answers.sql`
- `grade_mini_quiz_question_preview(question_id, submitted_answer)`
  - 익명 플레이 중 문제별 피드백을 받을 때 사용한다.
  - 정답 테이블은 직접 공개하지 않고, security definer 함수가 서버에서 채점한다.
- `claim_mini_quiz_stage_result_from_answers(chapter_id, stage_number, answers)`
  - 게스트/소셜 전환 후 sessionStorage에 있던 답안을 넘겨 서버에서 다시 채점하고 저장한다.
  - 클라이언트가 넘긴 `correct_count`, `star_count`, `earned_beans`는 신뢰하지 않는다.
  - 중복 보상은 `user_stage_progress.first_cleared_at`, `best_star_count` 기준으로 막는다.

`answers` 형식:

```json
[
  {
    "question_id": "git-start-init-mcq-01",
    "submitted_answer": "a",
    "submit_reason": "manual"
  }
]
```

주의:

- 기존 쿠키 기반 게스트는 `auth.uid()`가 없기 때문에 `auth.users`에 연결되는 stage progress를 저장할 수 없다.
- Supabase anonymous auth 게스트는 `auth.uid()`가 있으므로 기존 user progress, wallet, activity stats 테이블을 그대로 사용할 수 있다.
- 위 RPC는 Supabase authenticated 사용자 기준이다. Supabase anonymous auth 게스트도 authenticated role이므로 호출할 수 있다.

### Supabase anonymous auth 전환

- 게스트 시작은 `/auth/guest`에서 `supabase.auth.signInAnonymously()`를 호출한다.
- 랜덤 게스트 닉네임은 anonymous user metadata의 `name`으로 넣는다.
- `ensure_user_app_data()`가 anonymous user의 profile, wallet, activity row를 초기화한다.
- JWT의 `is_anonymous` claim 또는 `user.is_anonymous` 값으로 앱 안에서는 `authRole: "guest"`로 분류한다.
- anonymous guest는 Supabase Auth 기준으로는 `authenticated` role을 쓰므로 기존 RLS와 RPC의 `auth.uid()` 경로를 그대로 사용할 수 있다.
- 소셜 로그인 전환은 anonymous session이 있으면 일반 `signInWithOAuth()`가 아니라 `linkIdentity()`를 사용한다.
- Supabase 프로젝트 설정에서 Anonymous Sign-Ins와 Manual Identity Linking을 활성화해야 한다.

## 렌더링 전략

### 전체 원칙

- 공개 읽기 페이지는 가능하면 SSG 또는 ISR로 전환한다.
- 개인화, 보상, 기록 페이지는 SSR을 유지한다.
- 서버에 쓰는 행동은 Server Action 또는 Supabase RPC로 처리한다.
- 익명 플레이 상태는 클라이언트 sessionStorage에 둔다.
- 게스트/로그인 기록은 DB에 저장한다.

### 현재 구조의 제약

- 현재 `(app)/layout.tsx`가 쿠키, 게스트 상태, Supabase auth claims, 현재 유저, 공지사항을 매 요청마다 읽는다.
- 따라서 `(app)` 아래 페이지는 내용이 정적이어도 AppShell 때문에 SSR로 잡힌다.
- 공개 페이지를 진짜 SSG/ISR로 만들려면 개인화 레이아웃과 공개 레이아웃 분리가 필요하다.

### 추천 레이아웃 방향

```txt
app/
  page.tsx

  (public)/
    layout.tsx
    study/
    community/
    credits/

  (app)/
    layout.tsx
    lobby/
    profile/
    community/questions/new/
```

- `(public)` 레이아웃은 개인화 데이터를 읽지 않는다.
- `(app)` 레이아웃은 현재처럼 유저, 게스트, 공지, 프로필 메뉴 등 개인화 AppShell을 담당한다.
- route group만 다르므로 URL은 바뀌지 않는다.

### 페이지별 초안

- `/`: SSG
- `/credits`: SSG. 공개 레이아웃으로 분리하고 `force-static`을 명시한다.
- `/study`: 정적 학습 데이터 기반이면 SSG.
- `/study/[chapterId]/[stageId]`: 정적 문제 세트 기반이면 SSG + `generateStaticParams`.
- `/community/tips`: ISR 후보.
- `/community/notices`: ISR 후보.
- `/community/notices/[id]`: ISR 후보.
- `/community/questions`: 짧은 ISR 또는 SSR.
- `/community/questions/[id]`: 댓글/반응 최신성이 중요하므로 SSR 우선, 추후 짧은 ISR 검토.
- `/lobby`: 개인화 대시보드이므로 SSR.
- `/profile`: 개인화 페이지이므로 SSR.
- `/profile/[id]`: 공개 프로필 정책에 따라 SSR 또는 ISR 추후 결정.
- `/challenge`: 오늘 날짜와 보상 상태가 섞이면 SSR 우선.
- `/challenge/play`: 오늘의 문제와 완료 여부가 섞이면 SSR 또는 클라이언트 상태 기반.
- `/community/questions/new`: 작성 페이지이므로 세션 필요, SSR.

## 구현 순서 초안

1. 익명 상태 모델 추가
2. AppShell에서 익명 헤더 표시 지원
3. 공개 페이지 접근 예외 처리
4. 학습/챌린지 플레이 결과를 익명에게도 표시
5. 저장/보상/작성 순간 로그인/게스트 유도 모달 연결
6. sessionStorage pending result 저장
7. 게스트 시작 또는 소셜 로그인 후 pending result 반영
8. 커뮤니티 읽기 공개 및 작성/댓글/반응 보호
9. 공개 레이아웃과 개인화 레이아웃 분리
10. SSG/ISR 적용

## 질문지

남은 질문: 0개

1. 익명 헤더의 프로필 자리 문구와 형태는 `시작하기` 버튼으로 확정할까?
   - 결정: `시작하기` 버튼을 둔다.

2. 첫 방문 모달은 어떤 페이지에서 띄울까?
   - 후보: 모든 공개 페이지 첫 방문, 홈에서만, 앱 내부 공개 페이지에서만.
   - 결정: 첫 방문 모달은 띄우지 않고, 저장/보상/작성 행동 때만 모달을 띄운다.

3. 첫 방문 모달의 버튼은 무엇으로 둘까?
   - 후보: 게스트로 시작 / 로그인하기 / 둘러보기.
   - 결정: 저장 행동 모달에는 제목, 상황에 맞는 꼬꼬 이미지, 소셜 로그인, 게스트 로그인을 제공한다. `나중에` 버튼은 두지 않고, X 버튼 또는 바깥 클릭으로 닫는다.

4. 익명 스테이지 클리어 결과를 게스트 전환 후 바로 저장할까?
   - 후보: 현재 결과 저장, 저장하지 않음, 여러 결과 누적 저장.
   - 결정: 현재 클리어 결과만 저장한다.

5. 소셜 로그인 후 pending result 반영 시점은 언제로 할까?
   - 후보: auth callback 직후, AppShell 마운트 후 클라이언트 액션, 결과 페이지 복귀 후.
   - 결정: AppShell 마운트 후 클라이언트가 pending result를 감지해서 서버 액션으로 반영한다.

6. 익명 결과 임시 저장소는 sessionStorage로 확정할까?
   - 후보: sessionStorage, localStorage, 저장하지 않음.
   - 결정: sessionStorage를 사용한다.

7. 익명 사용자가 `/lobby`에 접근하면 어디로 보낼까?
   - 후보: lobby 표시, 홈으로 redirect, 시작 모달 표시.
   - 결정: `/lobby`를 익명에게도 보여준다. 이어하기와 개인 기록은 기록이 없을 때 자연스러운 빈 상태 UI로 처리하고, 저장/보상 영역은 안내 및 로그인/게스트 유도 모달로 연결한다.

8. `/profile/[id]`는 공개 프로필로 열어둘까?
   - 후보: 공개, 세션 필요, 본인/관리자만.
   - 결정: 공개 프로필 방향으로 간다. 다만 현재 단계에서는 게스트나 익명에게는 보이지 않게 유지한다.

9. 챌린지는 공개 레이아웃으로 옮길까, 개인화 레이아웃에 남길까?
   - 후보: 공개 체험 우선, 보상/완료 상태 우선.
   - 결정: 챌린지 소개와 플레이는 공개하고, 완료 저장·데일리 퀘스트 반영·보상 수령만 세션 필요로 처리한다.

10. 커뮤니티 질문 상세의 댓글/반응은 익명에게 어떻게 보여줄까?
    - 후보: 읽기만 가능, 버튼은 disabled, 버튼 클릭 시 모달.
    - 결정: 댓글/반응 버튼은 그대로 보여주고, 익명이 클릭하면 로그인/게스트 유도 모달을 띄운다.

11. SSG/ISR 레이아웃 분리는 이번 작업에 포함할까, 후속 작업으로 뺄까?
   - 후보: 이번 C안에 포함, 접근 정책 먼저 하고 후속으로 분리.
   - 결정: 같은 이슈 안에서 진행하되, 접근 정책과 익명 UX를 먼저 구현한 뒤 후속 커밋으로 공개/앱 레이아웃 분리와 SSG/ISR 적용을 진행한다.

12. 커스텀 Analytics 이벤트도 C안 작업에 포함할까?
    - 후보: 포함, 최소 포함, 제외.
    - 결정: 최소 포함한다. 익명에서 게스트/로그인으로 전환되는 핵심 지점만 추적하고, 개인정보는 기록하지 않는다.
